# 1. PRD — 제품 요구사항 문서

## 1.1 제품 개요

수업 실습 목적의 소규모 온체인 투표 시스템이다. 동시 접속 최대 5명, 총 투표수 30건 미만의 소규모 환경을 전제로 한다.  
이더리움 세폴리아 테스트넷의 스마트 컨트랙트를 기반으로 하며, 참여자는 크롬 브라우저와 MetaMask를 통해 웹에서 직접 투표에 참여한다.  
별도 백엔드 서버나 데이터베이스 없이 순수 온체인 아키텍처로 구현하여 구조를 단순하게 유지한다.

**투표 1회 = 컨트랙트 1개** 원칙을 따른다. 관리자는 웹 앱에서 직접 새 컨트랙트를 배포하여 투표를 생성하고, 투표 종료 후 다음 투표가 필요하면 새 컨트랙트를 다시 배포한다. Remix IDE 등 외부 도구는 필요하지 않다.

## 1.2 목적 및 배경

- 중앙화 시스템의 결과 신뢰성 문제를 블록체인의 불변성으로 해결
- 별도 회원가입 없이 지갑 주소만으로 참여 가능한 투표 환경 제공
- 투표 진행 중 실시간 현황, 종료 후 최종 결과를 누구나 온체인에서 검증 가능

## 1.3 이해관계자 및 사용자

| 역할 | 설명 | 권한 |
|------|------|------|
| **관리자 (Admin)** | 스마트 컨트랙트 배포자 | 후보자 등록, 투표 시작/종료 |
| **투표자 (Voter)** | MetaMask를 연결한 모든 사용자 | 투표 참여, 결과 조회 |
| **관람자 (Observer)** | 지갑 미연결 또는 투표 완료 사용자 | 결과 조회만 가능 |

## 1.4 핵심 기능 요약

| # | 기능 | 대상 |
|---|------|------|
| F-0 | 새 투표용 스마트 컨트랙트 배포 | 관리자 |
| F-1 | 후보자 등록 (이름 + 사진 URL) | 관리자 |
| F-2 | 투표 시작 / 수동 종료 | 관리자 |
| F-3 | MetaMask 연결 후 1인 1투표 | 투표자 |
| F-4 | 실시간 득표수 확인 (진행 중) | 누구나 |
| F-5 | 최종 득표수 및 당선자 확인 (종료 후) | 누구나 |

## 1.5 핵심 제약사항

| 항목 | 내용 |
|------|------|
| 네트워크 | Ethereum Sepolia Testnet 고정 |
| 브라우저 | Google Chrome (MetaMask 익스텐션 필수) |
| 후보자 등록 권한 | 컨트랙트 배포자(owner) 만 가능 |
| 투표 자격 | MetaMask 연결 시 누구나 (화이트리스트 없음) |
| 투표 종료 방식 | 관리자 수동 종료 (시간 자동 종료 없음) |
| 후보자 사진 저장 | 외부 공개 이미지 URL (imgur, GitHub raw 등 직접 임베드 가능한 URL만 허용. Google Drive 링크는 img 직접 임베드 불가하므로 사용 불가) |
| 투표 세션 | 컨트랙트 1개 = 투표 1회 (재사용 불가) |
| **운영 규모** | **동시 접속 최대 5명, 총 투표수 30건 미만 (수업 실습 전용)** |

## 1.6 규모 및 성능 전제

이 시스템은 수업 실습 환경을 전제로 설계되며, 다음을 명시적으로 **범위 밖**으로 정의한다.

| 항목 | 적용 안 함 (이유) |
|------|-----------------|
| 로드 밸런싱 / CDN | 동시 접속 5명 이하로 불필요 |
| 페이지네이션 | 후보자 수가 소수여서 단일 목록으로 충분 |
| 이벤트 구독(WebSocket) | 수동 새로고침 또는 폴링으로 충분 |
| 가스비 최적화 (배치 처리 등) | 총 트랜잭션 30건 미만으로 최소 비용 |
| 서버 인프라 / DB | 순수 온체인 + 정적 파일 서빙으로 대체 |

**채택 전략**: 정적 HTML/JS 파일을 GitHub Pages로 서빙하고, 모든 데이터는 컨트랙트에서 직접 읽는다. 별도 백엔드 없음.

## 1.7 성공 지표

- 중복 투표 시도 시 100% 스마트 컨트랙트 레벨에서 거부
- 투표 트랜잭션 완료 후 UI 득표수 갱신 (블록 확인 후 반영, 세폴리아 기준 약 12–15초)
- MetaMask 미설치·미연결·잘못된 네트워크 상황 모두 사용자 안내 제공

# 2. SRS — 소프트웨어 요구사항 명세

## 2.1 시스템 아키텍처 개요

**운영 규모 전제**: 동시 접속 ≤ 5명, 총 트랜잭션 ≤ 30건 (수업 실습)

```
[사용자 브라우저 (Chrome)]
        │
        ├── MetaMask 익스텐션 (트랜잭션 서명 / 지갑 관리)
        │
        ▼
[정적 Web Frontend]                    [브라우저 localStorage]
  HTML + CSS + Vanilla JS               - 마지막 배포 컨트랙트 주소
  ethers.js v6                          (DB 대체 — 별도 서버 불필요)
  컨트랙트 ABI + 바이트코드 번들
        │
        │  eth_call / eth_sendTransaction
        ▼
[Alchemy Sepolia RPC]  ──────────────▶  [스마트 컨트랙트 (Solidity)]
  (무료 티어, 안정적)                     - 투표 상태 (PREPARING/ONGOING/ENDED)
                                          - 후보자 목록 (이름, 사진URL, 득표수)
                                          - 투표 기록 mapping(address→bool)
                                          - 모든 데이터 영구 온체인 저장
```

### 아키텍처 확정 결정사항

| 계층 | 선택 | 결정 근거 |
|------|------|----------|
| **프론트엔드** | Vanilla HTML/CSS/JS | React/Vue 불필요 — 5명 규모에서 빌드 도구는 오버엔지니어링 |
| **Web3 라이브러리** | ethers.js v6 | 경량, 최신 API, MetaMask EIP-1193 호환 |
| **호스팅** | GitHub Pages | 무료, 정적 파일 서빙, 별도 서버 운영 불필요 |
| **RPC 노드** | Alchemy Sepolia 무료 티어 | 안정적인 공개 엔드포인트, 5명 규모에서 요청 한도 문제 없음 |
| **데이터베이스** | 없음 (온체인 대체) | 모든 투표 데이터는 스마트 컨트랙트에 저장 — DB 서버 불필요 |
| **세션/인증** | 없음 (MetaMask 대체) | 지갑 주소가 곧 신원 — 별도 인증 시스템 불필요 |
| **컨트랙트 주소 영속성** | 브라우저 localStorage | 새로고침 후에도 마지막 배포 주소 유지 — 별도 스토리지 서버 불필요 |
| **이미지 저장** | imgur / GitHub raw 파일 URL | Google Drive 공개 링크는 `<img>` 직접 임베드 불가(Google 정책). 직접 임베드가 되는 URL만 허용 |
| **득표수 갱신** | 10초 폴링 | WebSocket/이벤트 구독 불필요 — 5명 규모에서 RPC 부담 없음 |
| **컨트랙트 배포** | 프론트엔드에서 직접 배포 | ABI + 바이트코드를 JS에 번들 — Remix/Hardhat 등 외부 도구 불필요 |

## 2.2 기능 요구사항

### FR-00 컨트랙트 배포 (관리자 전용)

| ID | 요구사항 |
|----|---------|
| FR-00-1 | 관리자는 웹 앱 내 "새 투표 만들기" 화면에서 컨트랙트를 세폴리아 네트워크에 직접 배포할 수 있어야 한다. |
| FR-00-2 | 배포 시 MetaMask 서명 팝업이 나타나며, 관리자가 가스비를 지불하고 트랜잭션을 승인해야 한다. |
| FR-00-3 | 배포 트랜잭션 제출 후 "배포 중..." 상태를 UI에 표시해야 한다. |
| FR-00-4 | 배포 완료 후 생성된 컨트랙트 주소를 화면에 표시하고, 브라우저 `localStorage`에 자동 저장해야 한다. |
| FR-00-5 | 앱 진입 시 `localStorage`에 저장된 컨트랙트 주소가 있으면 SCR-00을 건너뛰고 마지막 배포 컨트랙트로 자동 연결해야 한다. 항상 가장 최근에 배포한 투표가 기본 연결 대상이다. |
| FR-00-6 | 일반 투표자는 `localStorage`에 주소가 없을 경우 두 가지 방법으로 투표에 연결할 수 있다: (1) 관리자가 공유한 URL(`?contract=0x...` 쿼리 파라미터 포함)로 접속 시 자동 연결, (2) SCR-00에서 주소 직접 입력. |
| FR-00-7 | 배포 완료 후 `?contract=컨트랙트주소` 쿼리 파라미터가 포함된 공유 링크를 자동 생성하여 복사 버튼과 함께 표시한다. 이 링크로 접속한 참여자는 주소를 별도로 입력하지 않아도 자동 연결된다. |
| FR-00-8 | 배포 실패 시 실패 원인(잔액 부족, 네트워크 오류 등)을 한국어로 안내해야 한다. |

### FR-01 지갑 연결

| ID | 요구사항 |
|----|---------|
| FR-01-1 | 사용자는 "지갑 연결" 버튼을 클릭하여 MetaMask로 지갑을 연결할 수 있어야 한다. |
| FR-01-2 | MetaMask가 설치되지 않은 경우 설치 안내 메시지와 공식 다운로드 링크를 표시해야 한다. |
| FR-01-3 | 세폴리아 네트워크가 아닌 다른 네트워크에 연결된 경우, 네트워크 전환 요청 UI를 표시해야 한다. |
| FR-01-4 | 연결된 지갑 주소가 컨트랙트 owner와 일치하면 관리자 UI를 노출해야 한다. |
| FR-01-5 | 지갑 연결 후 연결된 주소의 앞 6자리 + `...` + 뒤 4자리 형태로 주소를 헤더에 표시해야 한다. |
| FR-01-6 | 사용자가 MetaMask에서 지갑을 변경하거나 연결을 해제할 경우 UI가 즉시 반영되어야 한다. |

### FR-02 후보자 관리 (관리자 전용)

| ID | 요구사항 |
|----|---------|
| FR-02-1 | 관리자는 후보자 이름(텍스트)과 사진 URL(텍스트)을 입력하여 후보자를 등록할 수 있어야 한다. |
| FR-02-2 | 후보자 이름은 1자 이상 50자 이하이어야 한다. |
| FR-02-3 | 사진 URL은 http 또는 https로 시작하는 문자열이어야 한다. Google Drive 공개 링크는 `<img>` 직접 임베드가 불가하므로 허용하지 않는다. imgur, GitHub raw, 기타 직접 임베드 가능한 URL을 사용해야 한다. |
| FR-02-4 | 등록 전 입력된 URL로 이미지 미리보기를 표시해야 한다. 이미지 로드 실패 시 "이미지를 불러올 수 없습니다. 직접 임베드 가능한 URL을 사용하세요." 안내를 표시한다. |
| FR-02-5 | 후보자 등록은 투표 시작 전(준비 중 상태)에만 가능해야 한다. |
| FR-02-6 | 투표가 시작된 이후에는 후보자 추가 또는 삭제가 불가하며, 해당 버튼이 비활성화되어야 한다. |
| FR-02-7 | 후보자 목록은 등록 순(컨트랙트 인덱스 순)으로 표시된다. |
| FR-02-8 | 등록된 후보자 수가 2명 미만일 경우 투표 시작 버튼이 비활성화된다. |

### FR-03 투표 시작 / 종료 (관리자 전용)

| ID | 요구사항 |
|----|---------|
| FR-03-1 | 관리자는 후보자가 2명 이상 등록된 경우에만 투표를 시작할 수 있다. |
| FR-03-2 | 투표 시작 시 컨트랙트의 투표 상태가 `진행 중`으로 변경되고, 이후 후보자 등록이 차단된다. |
| FR-03-3 | 관리자는 진행 중인 투표를 수동으로 종료할 수 있다. |
| FR-03-4 | 투표 종료 후 컨트랙트 상태는 `종료`로 영구 변경되며, 재시작은 불가하다. |
| FR-03-5 | 투표 시작 및 종료 전 확인 다이얼로그를 표시해야 한다. ("투표를 시작하시겠습니까? 이후 후보자를 추가할 수 없습니다.") |

### FR-04 투표 참여

| ID | 요구사항 |
|----|---------|
| FR-04-1 | 투표는 투표 상태가 `진행 중`인 경우에만 가능하다. |
| FR-04-2 | 지갑이 연결된 사용자는 후보자 목록에서 1명을 선택하여 투표 버튼을 클릭할 수 있다. |
| FR-04-3 | 투표 버튼 클릭 시 MetaMask 서명 팝업이 나타나며, 사용자가 트랜잭션을 승인해야 한다. |
| FR-04-4 | 트랜잭션 제출 후 "트랜잭션 확인 중..." 상태를 UI에 표시해야 한다. |
| FR-04-5 | 이미 투표한 지갑 주소로 재투표 시도 시 스마트 컨트랙트가 거부하며, UI에 "이미 투표하셨습니다" 메시지를 표시해야 한다. |
| FR-04-6 | 트랜잭션 완료 후 해당 후보의 득표수가 갱신되어야 한다. |
| FR-04-7 | 투표 완료 후 해당 사용자의 투표 버튼이 비활성화되고 "투표 완료" 상태로 변경되어야 한다. |
| FR-04-8 | 지갑 미연결 상태에서 투표 시도 시 "지갑을 먼저 연결해주세요" 안내를 표시해야 한다. |

### FR-05 결과 조회

| ID | 요구사항 |
|----|---------|
| FR-05-1 | 투표 진행 중에는 각 후보자의 현재 득표수와 득표율(%)을 10초 주기로 갱신하여 표시한다. |
| FR-05-2 | 투표 종료 후에는 최종 득표수, 득표율, 당선자를 강조 표시한다. |
| FR-05-3 | 동률 발생 시 복수의 당선자를 동률로 표시하며 별도 처리 없이 그대로 노출한다. |
| FR-05-4 | 지갑 미연결 사용자도 결과 조회 화면에 접근할 수 있다. |
| FR-05-5 | 컨트랙트 주소는 화면 하단에 항상 표시되어, 사용자가 Etherscan에서 직접 검증할 수 있도록 링크를 제공한다. |

## 2.3 비기능 요구사항

| ID | 요구사항 |

| ID | 요구사항 |
|----|---------|
| NFR-03-1 | MetaMask 미설치, 네트워크 오류, 트랜잭션 실패 등 모든 오류 상황에 대해 한국어 안내 메시지를 제공해야 한다. |
| NFR-03-2 | 트랜잭션 제출부터 완료까지의 상태를 단계별로 UI에 표시해야 한다. |

| NFR-04-3 | 컨트랙트는 1회 배포로 1회 투표를 완료한다. 재사용이나 업그레이드 패턴은 적용하지 않는다. |
| NFR-04-4 | 후보자 사진은 `<img>` 태그에 직접 임베드 가능한 공개 URL(imgur, GitHub raw 파일 등)을 사용한다. Google Drive 공개 링크는 Google 정책상 직접 임베드가 불가하므로 허용하지 않는다. 별도 이미지 서버 구축은 필요하지 않다. |

## 2.4 스마트 컨트랙트 기능 명세

### 상태 및 데이터 구조

| 항목 | 타입 | 설명 |
|------|------|------|
| `owner` | `address public` | 컨트랙트 배포자 주소 (public 선언으로 자동 getter `owner()` 생성) |
| `votingStatus` | `enum` | `PREPARING` / `ONGOING` / `ENDED` |
| `candidates` | `Candidate[]` | 후보자 배열 |
| `Candidate.name` | `string` | 후보자 이름 |
| `Candidate.photoUrl` | `string` | 사진 URL |
| `Candidate.voteCount` | `uint256` | 득표수 |
| `hasVoted` | `mapping(address → bool) public` | 주소별 투표 여부 (public 선언으로 자동 getter 생성 — 별도 함수 불필요) |

### 함수 명세

| 함수명 | 권한 | 호출 가능 상태 | 설명 |
|--------|------|---------------|------|
| `addCandidate(name, photoUrl)` | owner | `PREPARING` | 후보자 등록 |
| `startVoting()` | owner | `PREPARING` (후보자 ≥ 2) | 투표 시작 |
| `endVoting()` | owner | `ONGOING` | 투표 수동 종료 |
| `vote(candidateId)` | 누구나 | `ONGOING` | 투표 실행 (중복 차단) |
| `getCandidates()` | 누구나 | 항상 | 후보자 목록 + 득표수 반환 |
| `getVotingStatus()` | 누구나 | 항상 | 현재 투표 상태 반환 |
| `hasVoted(address)` | 누구나 | 항상 | 특정 주소 투표 여부 반환 (public mapping 자동 getter) |
| `owner()` | 누구나 | 항상 | owner 주소 반환 (public address 자동 getter) |

### 이벤트 명세

| 이벤트 | 파라미터 | 발생 조건 |
|--------|---------|----------|
| `CandidateAdded` | `candidateId`, `name`, `photoUrl` | 후보자 등록 시 |
| `VotingStarted` | — | 투표 시작 시 |
| `VotingEnded` | — | 투표 종료 시 |
| `Voted` | `voter` (address), `candidateId` | 투표 완료 시 |

### 오류 조건 (Revert 사유)

| 조건 | 오류 메시지 |
|------|------------|
| owner가 아닌 주소가 관리자 함수 호출 | `"Not owner"` |
| `PREPARING` 아닌 상태에서 후보자 등록 | `"Voting already started"` |
| 후보자 2명 미만에서 투표 시작 | `"Need at least 2 candidates"` |
| `ONGOING` 아닌 상태에서 투표 | `"Voting is not ongoing"` |
| 이미 투표한 주소가 재투표 | `"Already voted"` |
| 존재하지 않는 후보자 ID | `"Invalid candidate"` |

**구성 요소 상세**

| 요소 | 설명 |
|------|------|

|------|------------|----------|

| 준비 중 (PREPARING) | 투표 준비 중 | 비활성 |
| 진행 중 · 미투표 | 투표하기 | 활성 |
| 진행 중 · 투표 완료 | ✅ 투표 완료 | 비활성 |
| 투표 종료 | — | 결과 화면으로 전환 |

## 4.3 Phase 1 — 스마트 컨트랙트 개발

**목표**: 온체인 로직 전체를 Solidity로 구현하고 Sepolia에서 검증한다.

### 구현 범위

| 구현 항목 | 관련 요구사항 |
|----------|-------------|
| `Candidate` struct 및 상태 변수 선언 (`owner`, `votingStatus`, `candidates[]`, `hasVoted`) | 2.4 상태 및 데이터 구조 |
| `onlyOwner` modifier 구현 | NFR-01-2 |
| `addCandidate(name, photoUrl)` 함수 | FR-02-1, FR-02-5 |
| `startVoting()` 함수 (후보자 ≥ 2 검증 포함) | FR-03-1, FR-03-2 |
| `endVoting()` 함수 | FR-03-3, FR-03-4 |
| `vote(candidateId)` 함수 (중복 투표 차단 포함) | FR-04-1 ~ FR-04-6 |
| `getCandidates()` view 함수 | FR-05-1, FR-05-2 |
| `getVotingStatus()` view 함수 | 2.4 함수 명세 |
| 이벤트 선언 (`CandidateAdded`, `VotingStarted`, `VotingEnded`, `Voted`) | 2.4 이벤트 명세 |
| ABI + 바이트코드 추출 | Phase 3 사전 준비 |

### 코드 리뷰 체크리스트

- [ ] 모든 상태 변경 함수에 적절한 modifier (`onlyOwner`, 상태 검증) 적용 여부
- [ ] `require` 오류 메시지가 2.4 오류 조건 표의 메시지와 일치하는지
- [ ] `hasVoted`와 `owner`가 `public`으로 선언되어 자동 getter 생성되는지
- [ ] `getCandidates()`가 `Candidate[] memory`를 반환하는지
- [ ] 재진입 공격(reentrancy) 가능성 없는지 (이 컨트랙트는 ETH 송수신 없으므로 해당 없음)
- [ ] 불필요한 상태 변수 없는지

### 검증 항목

- [ ] FR-03-1: 후보자 1명 상태에서 `startVoting()` 호출 시 revert 되는가
- [ ] FR-04-5: 동일 주소 두 번 `vote()` 호출 시 revert 되는가
- [ ] NFR-01-2: owner 외 주소가 `addCandidate()` 호출 시 revert 되는가
- [ ] FR-03-4: `endVoting()` 후 `startVoting()` 재호출 시 revert 되는가

### 테스트 케이스

| TC | 시나리오 | 기대 결과 |
|----|---------|----------|
| TC-01 | owner가 후보자 2명 등록 후 `startVoting()` 호출 | 성공, 상태 ONGOING |
| TC-02 | 후보자 1명만 등록 후 `startVoting()` 호출 | revert: "Need at least 2 candidates" |
| TC-03 | ONGOING 상태에서 `vote(0)` 호출 | 성공, 후보자 0번 득표수 1 증가 |
| TC-04 | 동일 주소로 `vote()` 재호출 | revert: "Already voted" |
| TC-05 | owner 외 주소로 `addCandidate()` 호출 | revert: "Not owner" |

**전체 E2E 시나리오 (Phase 7 최종 검증)**

```
1. GitHub Pages URL 접속 (localStorage 비어있음)
2. SCR-00에서 지갑 연결 후 새 컨트랙트 배포
3. 공유 링크 복사 → 다른 브라우저 탭에서 접속 → SCR-01 자동 이동
4. 관리자: 후보자 2명 등록 (각각 다른 사진 URL)
5. 관리자: 투표 시작
6. 투표자1 (지갑A): SCR-01에서 후보자 A 투표
7. 투표자2 (지갑B): SCR-01에서 후보자 B 투표
8. 투표자1 재투표 시도 → 차단 확인
9. 관리자: 투표 종료
10. SCR-03 자동 전환, 결과 확인
11. 관리자: "새 투표 만들기" → SCR-00으로 이동
```

### 사이드 이펙트 검증

- Phase 3의 `localStorage` 초기화가 "새 투표 만들기" 시 동작하는가
- 모든 Phase의 오버레이(SCR-04, SCR-05)가 SCR-03에서도 동작하는가
- GitHub Pages 배포 후 CORS 없이 Alchemy RPC 호출이 가능한가
